**Notes for next person:**
- The v2 storage header is 17 bytes (vs 16 for v1) due to 9-byte TMSP2 timestamp.
- HTYP2 is 32-bit BE; byte 0 bits map to u32 bits 24-31. See protocol.rs for details.
- Extension header fields are ordered by HTYP2 flag position. WACID covers both
  APID and CTID (each length-prefixed). WSID is 4 bytes fixed (no length prefix).
- The `Dlt::open()` currently silently skips v1 messages (returns 0 rows for v1 files).

### 2026-06-01 — Issue #64: v1 LEN-driven framer

**What was done:**
- Created `src/dlt/v1/framer.rs` with `Frame` struct and `scan_frames(data, file_index)`
  function.
- `Frame` contains: `storage_timestamp_ns` (u64), `storage_ecu` ([u8; 4]),
  `msg_start` (usize, offset of standard header), `msg_len` (usize, from LEN).
- Frame boundaries determined by the LEN field in the v1 standard header (bytes
  2-3, big-endian), not by scanning for the next `DLT\x01` marker. This ensures
  false `DLT\x01` sequences in payload data do not cause mis-framing.
- `storage_timestamp_ns` computed as `seconds * 1_000_000_000 + microseconds * 1_000`.
- Emits `InvalidVersion` when HTYP bits 5-7 are not 1, then resyncs.
- Emits `Truncated` when data ends before complete storage + standard header.
- Emits `LengthMismatch` when declared LEN exceeds available data or is below
  minimum standard header size.
- On any error, resyncs to the next `DLT\x01` marker.
- Uses shared `ParseError`/`ParseErrorKind` from `dlt/error.rs`.
- Registered `pub mod framer` in `v1/mod.rs`.

**Tests added (9 new, 100 total passing, 10 v1 legacy ignored):**
- `finds_single_v1_frame`, `finds_multiple_frames`, `empty_input`,
  `truncated_storage_header`, `truncated_mid_header`, `wrong_version_resyncs`,
  `length_mismatch_declared_exceeds_available`,
  `false_marker_in_payload_not_confused_as_frame`,
  `file_ending_mid_message_after_valid`.
